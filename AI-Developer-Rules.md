# AI-Assisted Development Guidelines

## 1. Purpose and Applicability

This document defines the guidelines that every developer should follow when using AI coding assistants to produce, modify, or review code. These are strongly recommended practices for any AI-assisted development workflow.

## 2. Core Principle: The Developer Decides, the Developer Owns

AI assistants are execution tools. They do not make architectural decisions, choose implementation approaches, or decide what to build. While you should collaborate with an AI to explore options, generate alternatives, or draft solutions, every decision should be reviewed and approved by you, the developer. The AI proposes; you decide. No decision is final until a human developer has explicitly approved it and documented that approval in the plan.

You own every line of code you submit. When you commit code to a repository or submit a pull request, you are signing off that the code is correct, secure, and meets the team's standards. It does not matter whether you wrote the code by hand or an AI generated it. The act of committing is your personal certification that you reviewed, understood, and approved every change. "The AI wrote it" or "the AI suggested it" is never an acceptable explanation for a bug, a vulnerability, a design flaw, or a missed requirement. You approved it. You own it.

If the AI had to assume something in order to produce its output, your plan was not detailed enough. Go back and fix the plan.

## 3. Guideline 1: Create a Plan File Before Development Starts

No code should be generated by an AI assistant until a written plan file exists for the work being done.

### 3.0 Exception for Trivial Changes

Small tweaks, typo fixes, minor formatting corrections, and other trivial changes do not require a plan file. However, the developer should clearly note in the commit message that the change was AI-assisted and describe what was changed and why. All other rules in this document, including developer review, ownership, and accountability, should still apply. If there is any doubt about whether a change qualifies as trivial, create the plan file.

### 3.1 Building the Plan File

Every non-trivial task (whether it is a new feature, a bug fix, or a refactor) should have a Markdown plan file. The developer should build this plan collaboratively with their AI assistant through conversation. Start by describing what you want to build, then let the AI help you draft and structure the plan document through a back-and-forth discussion. As the plan takes shape, ask the AI to help improve it, poke holes in it, surface assumptions, identify gaps, and flag anything that is ambiguous or underspecified. The goal is for both the developer and the AI to agree that the plan contains enough detail and specific coverage to begin development, and that the AI will not need to make any assumptions or decisions without the developer's approval. The plan is not ready until both sides confirm this.

### 3.2 Plan Files for New Applications

When building a new application, an additional architecture document is recommended. This should cover the technology stack with rationale for each choice, the high-level component and module structure, data models and storage design, API or integration design, and deployment considerations. Keep the architecture document separate from individual feature plans so it can be referenced across all phases of development.

### 3.3 Where Plan Files Live

Plan files should be committed to the repository in a dedicated /plans/ directory somewhere in the repository so they are version-controlled and accessible to reviewers. Plan files are living artifacts and should be updated when decisions change.

## 4. Guideline 2: The Developer Should Approve the Final Plan Before Development Begins

The AI may help build and refine the plan, but the final version should be explicitly approved by the developer.

Before moving to development, read the complete plan one final time and confirm that every detail reflects your decisions and not the AI's preferences. During collaborative drafting, subtle drift can occur. Priorities shift, phrasing gets softened, or a detail you cared about gets watered down. A full read-through confirms the document still matches your intent. You are signing off on this plan as the specification that will govern the code. Any deviation from the plan during development is your responsibility to catch and correct.

The plan is complete when a developer (human or AI) could implement it without needing to make a single assumption. If any part of the plan requires the implementer to guess, decide, or infer intent, it is not ready.

## 5. Guideline 3: Development Should Follow Structured, Scoped Phases

For any task that spans more than a single focused session, the work should be broken into discrete phases before development begins.

### 5.1 AI Creates the Phase Breakdown

Using the approved plan file, ask the AI to break the work into manageable, self-contained phases. Each phase should have a clear start point, end point, and defined deliverables. The AI should verify that each phase builds on the outputs of the previous phase and that they are sequenced in the correct build order. No phase should assume something exists that has not been built in a prior phase. Circular dependencies or missing prerequisites should be identified and resolved as part of this step.

### 5.2 Developer Reviews and Approves the Phases

Once the AI has produced the phase breakdown, the developer should review the proposed phases and confirm that the sequencing is logical, the scope of each phase is reasonable, and nothing has been missed or reordered in a way that creates risk. The developer approves the final phase breakdown before any development begins. If something is off, collaborate with AI to revise until you are satisfied.

### 5.3 Phase Documentation

Each phase should have a brief document (or a clearly delineated section of the plan file) that includes what is being built in this phase, what prior outputs it depends on, and the acceptance criteria for considering the phase complete. This document should be self-contained enough that you could start a fresh AI session, provide only this document and the relevant existing code, and begin work cleanly.

### 5.4 Fresh Context Per Phase

When starting a new phase, begin a new AI session with a clean context window. Load only the phase document, the architecture document (if applicable), and the relevant existing code. Do not carry forward conversation history from prior phases. AI models have a finite context window, and when that window fills up, the model compacts its memory by discarding or summarizing older content. This compaction process can cause the AI to lose critical details from your plan, requirements, or earlier conversation. When those details are lost, the AI does not stop and ask. It fills in the gaps on its own, often generating code based on incorrect assumptions or fabricated details (commonly referred to as hallucination). Starting each phase with a clean, focused context prevents this problem and ensures the AI is working from accurate, complete information.

## 6. Guideline 4: All Code Should Pass a Four-Layer Review Before It Can Be Merged

Code should not be merged into the codebase until it has passed all four review layers: (1) the development AI assistant, (2) the developer, (3) a third-party AI code review service, and (4, if possible) a peer developer code review at merge time. All four are strongly recommended.

### 6.0 Rules That Apply to All Review Layers

Clear the context first. Before any AI code review (Layer 1 or Layer 3), clear the context window (e.g., /clear). A review performed without clearing context is not ideal. The AI needs a fresh pair of eyes, not the assumptions (lingering context) it built up during development.

Fix, re-review, repeat. When any review layer finds issues, fix them and run that review again. Repeat until it passes clean. A single pass that finds problems is never sufficient.

Use your judgment on findings. AI reviewers will sometimes flag intentional code as a problem. The developer should evaluate every finding and either fix the code or document why the finding does not apply. Blindly accepting findings is as unacceptable as blindly ignoring them.

### 6.1 Layer 1: Development AI Code Review

Once implementation is complete, clear the context window and ask the AI to review the code against the following checklist:

- Correctness and Logic. Does the code do what the plan says it should? Any logic errors, off-by-one mistakes, unhandled edge cases, or incorrect assumptions?
- Security and Error Handling. Are inputs validated? Any injection risks, exposed secrets, or missing auth checks?
- Maintainability and Standards. Is the code clean, readable, and consistent with project conventions? Any unnecessary complexity or duplicated logic?
- Performance. Any obvious bottlenecks, unnecessary database calls, unbounded loops, or inefficient algorithms?
- Documentation. Are code comments and project documentation accurate and up to date?
- Unit Test Coverage. Do tests exist for all new and modified code, covering success paths, failure paths, and plan file edge cases? Are the tests meaningful?

See [CODEREVIEW.md](CODEREVIEW.md) for detailed checklists, prompt templates, and review instructions.

### 6.2 Layer 2: Developer Code Review

After the AI review cycle is complete with no outstanding issues, the developer should perform their own manual review of the final code. This should not be optional. You are the person who understands the intent behind the code and the broader project context. Read through every file being changed and verify that the code matches the plan, follows conventions, and does not introduce anything unexpected.

You should be able to explain every line of code in the changeset. If you cannot explain what a block does, why it exists, or what edge cases it handles, do not move forward. If your review surfaces issues, fix them and repeat the Layer 1 AI review cycle before continuing. By completing this review and moving forward, you are personally certifying that the code is ready for production. From this point on, this is your code, not the AI's.

### 6.3 Layer 3: Third-Party AI Code Review

After the developer has completed their own review and the pull request has been submitted, a third-party AI code review service performs an automated review. This occurs either locally through the service's IDE integration prior to the PR or through its integration with your repository hosting platform as part of the pull request workflow.

For each issue the service identifies:

1. The developer reviews the finding to determine if it is a legitimate issue or a false positive (per Section 6.0).
2. If legitimate, feed the issue back to the development AI assistant to fix.
3. Push a new commit with the fix.
4. Run the third-party review again.
5. Repeat until the review comes back clean or all remaining findings have been documented as intentional by the developer.

### 6.4 Layer 4: Peer Developer Code Review at Merge

After Layers 1 through 3 have passed, a separate developer (not the author of the code) should perform a peer code review before the pull request can be approved and merged. This is the final human gate. The peer reviewer should verify that the code matches the plan file, that the tests are substantive and not superficial, and that the code does not show signs of unreviewed AI output (e.g., unnecessary boilerplate, inconsistent naming, patterns that do not match the project's conventions). Where possible, the PR should not be merged until the peer reviewer has approved it.

Only after all four review layers have passed and all findings have been resolved or documented may the code be merged.

### 6.5 Code Review Instruction File (Recommended)

Developers are encouraged to maintain a CODEREVIEW.md file in the repository that contains the standard prompts, checklists, and focus areas used during AI code reviews. This ensures consistency across the team and makes it easy to onboard new developers into the review process. This file should only be loaded into AI sessions during review, not during development, to avoid consuming context unnecessarily.

See [CODEREVIEW.md](CODEREVIEW.md) for details

## 7. Guideline 5: Unit Tests Should Be Created or Updated with Every Change

No code change should be submitted in a pull request without corresponding unit tests. This is one of the most important practices in AI-assisted development.

### 7.1 New Code Requires New Tests

Every new function, method, endpoint, or component introduced by AI-assisted development should have unit tests that verify its behavior. Tests should cover the success path, the failure path, and the edge cases documented in the plan file. Aim for full code coverage.

### 7.2 Modified Code Requires Updated Tests

If existing code is modified, the corresponding tests should be updated to reflect the new behavior. If the modified code did not previously have tests, tests should be written as part of the change. There should be no exemptions for "legacy code" or "small changes."

### 7.3 Test Quality Standards

Tests generated by AI should be reviewed for quality. The following are not acceptable:

- Superficial assertions that always pass regardless of the code's behavior
- Tests that mirror the implementation rather than verifying correct results for given inputs
- Missing edge case coverage when edge cases are documented in the plan file

These quality checks should be codified in a [CODEREVIEW.md](CODEREVIEW.md) file (see Section 6.5) so they are consistently applied during every code review.

### 7.4 Tests Should Pass Before PR Creation

All existing and new tests should pass before a pull request is created. A PR with failing tests should be rejected. No exceptions.

### 7.5 Integration and End-to-End Tests

Unit tests are the minimum requirement per change. For larger features or multi-phase work, integration tests and end-to-end tests should be planned as a dedicated part of each phase and should be completed before the feature is considered finished.

## 8. Guideline 6: Pull Request Standards

A pull request is the final gate before code enters the codebase. It should meet every requirement below to be eligible for merge.

### 8.1 PR Checklist

Before submitting a pull request, the developer should confirm all of the following:

- [ ] A plan file exists for this work and is committed to the repository. (if a substantial change)
- [ ] The plan file was validated with an AI assistant and refined until no assumptions remain.
- [ ] The AI assistant performed a structured code review on a fresh context and all findings were addressed.
- [ ] I performed my own manual code review of the final changeset.
- [ ] I can explain every line of code in this PR.
- [ ] Unit tests exist for all new and modified code.
- [ ] All tests pass in the CI pipeline.
- [ ] No secrets, PII, or proprietary data were provided to the AI tool during development.
- [ ] The code conforms to the project's coding standards, linting rules, and architectural patterns.

### 8.2 PR Description Requirements

Every pull request should include a description of all changes being introduced and a link to the plan file used for the work.

## 9. Data Security Rules for AI Tool Usage

These rules apply to every interaction with an approved AI coding assistant, whether during planning, development, or review.

### 9.1 Never Provide to an AI Tool

- API keys, tokens, passwords, certificates, SSH keys, or connection strings
- Personally identifiable information (PII), protected health information (PHI), or financial data
- Internal network details: IP addresses, hostnames, firewall rules, security configurations
- Trade secrets or proprietary algorithms that represent competitive advantages (unless the tool is under an approved data protection agreement)

### 9.2 Secrets Scanning

All repositories should have pre-commit hooks or CI checks for secrets detection. AI-generated code is particularly prone to including placeholder credentials or example keys that may resemble real secrets. Secrets scanning is strongly recommended and applies to all commits, regardless of origin.

## 10. When Things Go Wrong

Not every task will go according to plan. When problems surface during AI-assisted development, the severity determines the response. But the core principle is always the same: stop, document what went wrong, and fix the plan before fixing the code.

### 10.1 Minor Drift

The AI produces code that technically works but does not match the plan's intent. This could be a wrong pattern, an over-engineered solution, or a misunderstood requirement. Stop immediately. Do not let the AI build more code on top of a misaligned foundation. Clarify the intent, update the plan if needed, and redo the work.

### 10.2 Phase-Level Failure

You reach the end of a phase and the approach fundamentally does not work. Do not patch it in place. Stop the phase, document what failed and why, update the plan, and restart the phase with a clean AI session and the corrected plan.

### 10.3 Plan-Level Failure

Development reveals a fundamental flaw in the architecture or overall plan. Pause all development. Document the failure thoroughly. Revise the architecture and plan documents. Re-validate the phase breakdown. Resume from the earliest affected phase.

### 10.4 Failure Log

Maintain a brief log of what went wrong on each occurrence: what broke, why, and what was changed. Over time, patterns will emerge that help you refine your planning process and avoid repeating the same mistakes.

## 11. Accountability and Enforcement

Every commit is a personal sign-off. When a developer commits code and submits a pull request, they are accepting full accountability for that code. The development process in this document (planning, validation, AI review, developer review, and testing) exists to ensure that by the time code reaches a PR, the developer has thoroughly vetted it. There is no scenario in which responsibility for a defect, vulnerability, or outage can be attributed to the AI tool. The developer reviewed it, approved it, and submitted it. The developer is accountable.

Pull requests that do not meet the requirements in this document should be rejected by reviewers.
